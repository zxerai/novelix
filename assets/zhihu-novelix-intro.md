# 如何用 AI 自动写小说？我做了一个 10 Agent 协作系统，13 天写了 12 万字

> 前言：这是一个关于"用 AI 写小说"的技术实践分享。我会从踩坑经历讲起，详细介绍 Novelix 的架构设计、核心能力和实际效果。如果你对 AI 写作感兴趣，或者想了解多 Agent 协作的实现方式，这篇文章应该能给你一些启发。

---

## 一、问题：为什么直接用 ChatGPT 写不好小说？

去年我尝试用 ChatGPT 写一本玄幻小说。

写到第 5 章，发现主角的名字变了两次。

写到第 8 章，一个已经死掉的角色又活了。

写到第 10 章，前面埋的伏笔全忘了。

通篇都是"仿佛"、"不禁"、"宛如"——一看就是 AI 写的，读者直接弃书。

**我复盘了一下，发现问题的根源不是 AI 能力不行，而是架构设计有问题。**

你让一个 AI 模型写十万字，它需要：

1. 记住前面所有的剧情
2. 保证角色设定不冲突
3. 维护伏笔和悬念
4. 控制叙事节奏
5. 保持文风一致

这些任务对单个模型来说太重了。即使你给它很长的上下文窗口，它也无法真正"理解"这些信息之间的关系。

**本质上，你在要求一个模型同时扮演编剧、作家、编辑、校对四个角色，而且每个角色都要做到专业水准。**

这不现实。

---

## 二、思路：用多 Agent 协作替代单模型生成

我是个程序员，想不明白的事就写代码解决。

我的思路是：

**既然一个 Agent 做不好，那就用 10 个 Agent 分工协作。**

这就像拍电影——导演、编剧、演员、剪辑师各司其职，而不是让一个人包办所有工作。

我做了一个叫 **Novelix** 的工具，它的核心架构是：

```
10 Agent Pipeline:
  1. Architect（架构师）— 搭建世界观、角色设定、主线大纲
  2. Planner（规划师）— 把大纲拆成每章的情节点
  3. Writer（作家）— 正式写作
  4. Auditor（审计员）— 检查 33 个维度的连续性
  5. Reviser（修订师）— 根据审计结果修改
  6. QualityChecker（质检员）— 最终质量检查
  7. StyleAnalyzer（风格师）— 文风一致性
  8. Radar（雷达）— 扫描外部素材
  9. Analyst（分析师）— 数据分析
  10. Reviewer（审核员）— 人工审核门控
```

每个 Agent 都有独立的 prompt、独立的上下文、独立的职责。

**关键设计：Agent 之间通过"真相文件"通信，而不是通过上下文窗口。**

---

## 三、核心设计：33 维连续性审计

这是 Novelix 最核心的能力。

每写一章，审计员会对照 7 个"真相文件"检查 33 个维度：

### 真相文件（7 个）

| 文件 | 内容 |
|------|------|
| `character_matrix.md` | 角色设定（性格、外貌、能力、关系） |
| `plot_threads.md` | 伏笔和悬念追踪 |
| `world_state.md` | 世界状态（地点、势力、规则） |
| `inventory.md` | 物资清单（装备、道具、资源） |
| `timeline.md` | 时间线 |
| `emotional_arcs.md` | 情感弧线 |
| `style_guide.md` | 文风指南 |

### 审计维度（33 个）

- 角色位置连续性
- 角色状态连续性
- 物资连续性
- 伏笔回收检查
- 大纲偏离检测
- 叙事节奏分析
- 情感弧线合理性
- 对话风格一致性
- ……（共 33 个）

**审计员会生成一份审计报告，列出所有问题，然后修订师根据报告修改。**

如果审计不通过，这一章会打回重写，直到通过为止。

**这就是为什么 Novelix 写的小说不会出现"角色穿帮"和"伏笔遗忘"的问题。**

## 四、核心设计：去 AI 味

AI 写的小说有一个很明显的问题：

**太"干净"了。**

- 句式太工整
- 用词太书面
- 节奏太均匀
- 情感太克制

读者一眼就能看出来是 AI 写的。

Novelix 内置了 22 条改写规则，专门解决这个问题：

### 词汇层面

- 消灭 AI 高频词：仿佛、不禁、宛如、突然、瞬间、顿时
- 替换为更自然的表达

### 句式层面

- 长短句交替
- 倒装句、省略句
- 口语化表达

### 段落层面

- 段落呼吸（长短段落交替）
- 情绪外化（用动作和环境描写代替直接描述）

### 朱雀专项

- 句长波动
- 标点多样性
- 代词降频
- 避免完美过渡
- 副词降频

**改完之后，朱雀（AI 检测工具）的检测率从 90% 降到 10% 以下。**

## 五、核心设计：多模型路由

Novelix 支持为不同的 Agent 配置不同的模型：

```bash
# Writer 用 Claude（创意强）
novelix config set-model writer claude-sonnet-4-6

# Auditor 用 GPT-4o（快速便宜）
novelix config set-model auditor gpt-4o

# Radar 用本地模型（零成本）
novelix config set-model radar ollama/llama3
```

**为什么要这么做？**

因为不同的任务对模型的要求不同：

| Agent | 需求 | 推荐模型 |
|-------|------|----------|
| Writer | 创意、文采 | Claude、GPT-4o |
| Auditor | 逻辑、准确 | GPT-4o、Gemini |
| Planner | 结构、规划 | Claude、GPT-4o |
| Radar | 扫描、检索 | 本地模型（零成本） |

**这样可以用最低的成本获得最好的效果。**

## 六、实际效果：13 天写了 12 万字

我用 Novelix 写了一本叫《赘婿仙帝》的仙侠小说。

### 数据

| 指标 | 数据 |
|------|------|
| 写作天数 | 13 天 |
| 总字数 | 12 万字 |
| 章节数 | 13 章 |
| 平均每章 | 9200 字 |
| 角色穿帮 | 0 次 |
| 伏笔遗忘 | 0 次 |
| AI 检测率 | < 10% |

### 工作流程

每天的工作流程是这样的：

1. **早上**：打开电脑，看看昨晚 AI 写的章节
2. **审核**：花 10 分钟看看有没有问题
3. **上班**：该干嘛干嘛
4. **晚上**：回来发现又多了 9000 字

**我每天只需要花 10 分钟审核，剩下的时间都是 AI 在自动工作。**

### 成本

| 模型 | 用途 | 成本 |
|------|------|------|
| Claude Sonnet | Writer | ~$0.5/章 |
| GPT-4o | Auditor | ~$0.1/章 |
| 本地模型 | Radar | $0 |
| **总计** | | **~$0.6/章** |

13 章 × $0.6 = **$7.8（约 55 元人民币）**

**55 元写了 12 万字，平均每千字 0.46 元。**

## 七、怎么用？

### 安装

```bash
npm i -g @actalk/novelix
```

### 初始化

```bash
novelix init my-novel
cd my-novel
```

### 配置模型

```bash
novelix doctor  # 检查环境
novelix config set-global --provider openai --base-url <url> --api-key <key> --model <model>
```

### 创建书籍

```bash
novelix book create --title "我的小说" --genre xuanhuan --platform tomato
```

### 开始写作

```bash
novelix write next 我的小说 --count 10  # 连写 10 章
```

### Studio 工作台

```bash
novelix  # 打开 http://localhost:4567
```

Studio 提供：
- 书籍管理
- 章节审阅
- 数据分析
- 知识图谱
- 模型配置

## 八、技术细节（给程序员看的）

### 技术栈

- **语言**：TypeScript
- **运行时**：Node.js 22+
- **包管理**：pnpm
- **前端**：React + Vite
- **后端**：Hono
- **测试**：Vitest

### 项目结构

```
novelix/
├── packages/
│   ├── core/          # 核心库（Agent、Pipeline、StateManager）
│   ├── cli/           # CLI 工具
│   └── studio/        # Web 工作台
├── books/             # 书籍数据
├── genres/            # 题材配置
└── skills/            # 技能配置
```

### Agent Pipeline 流程

```
Book.create()
  → Architect.init()
  → Planner.plan()
  → Writer.write()
  → Auditor.audit()
  → Reviser.revise()
  → QualityChecker.check()
  → Reviewer.approve()
  → Chapter.save()
```

### 真相文件机制

每个 Agent 执行前，会读取 7 个真相文件作为上下文。

执行后，会更新相关的真相文件。

**这样就实现了"跨章节记忆"，而不需要依赖模型的上下文窗口。**

## 九、局限性和改进方向

### 当前局限

1. **创意依赖模型**：Novelix 只是工具，创意还是来自作者
2. **需要人工审核**：虽然错误率很低，但还是需要人工把关
3. **成本可控但不为零**：长篇小说的成本会线性增长

### 改进方向

1. **更强的 Agent 协作**：引入更多 Agent，实现更细粒度的分工
2. **本地模型支持**：降低对云端 API 的依赖
3. **自动审核**：用 AI 审核 AI，进一步减少人工介入

## 十、总结

**Novelix 不是"AI 写小说"，而是"AI 帮你写小说"。**

区别在于：

- "AI 写小说"：你给一个主题，AI 自己写，你不管
- "AI 帮你写小说"：你负责构思，AI 负责执行，你审核

**灵魂是你的，AI 只是笔。**

你本来就有写一本百万字小说的能力，只是缺一个帮你把想法落地的工具。

现在，工具来了。
